Read this first. Self-hosted Path of Exile 2 (PoE2) browser utility suite: pricing, crafting, rune/drop valuation, and an in-progress gear search. Local-first.

## Stack
- Zero-dependency Node.js — built-in http/fs/path/child_process only. No framework, no npm deps.
- Single server.js + standalone vanilla HTML/CSS/JS pages (index.html, character-upgrades.html, craft-optimizer.html).
- Run locally: node server.js -> http://127.0.0.1:17777.

## Data sources
- **GGG Trade2 is the source. poe.ninja stays banned EXCEPT one scoped, user-approved exception (2026-06-25, "on probation"):** currency-exchange-ratio items (Verisium **alloys**, chase orbs) that GGG's offer API genuinely CANNOT price — proven: their exalted ask, exalted bid, divine side, and item-search are ALL placeholder noise (1ex / exactly 1 div=319ex / 0.5 div), decoupled from the real per-item ratio (Celestial Alloy real ~160ex). That ratio only exists in GGG's computed currency-exchange (OAuth-gated cxapi) or poe.ninja. poe.ninja's PoE2 API (`poe.ninja/poe2/api/economy/currencyexchange/overview?leagueName=…&overviewName=Currency`) is **Cloudflare-blocked server-side** (404 + cf-ray for every league, even the active one) — so we do NOT call it from the server. Instead: a **currency-override store** (`.currency-overrides.json`, `CURRENCY_OVERRIDE_SEED` in server.js) the Rune Picker checks first, refreshed via a **browser snippet** the user runs ON poe.ninja (CF-cleared, same-origin) that copies the overview; they paste it → `POST /api/currency-overrides`. Still NO server-side poe.ninja calls. Everything else stays Trade2-only. Snippet caveat: reads `primaryValue` as the ex price (unverified — server can't reach poe.ninja to confirm the JSON shape); sanity-check a known item on first use. This is narrow + on probation — do NOT broaden poe.ninja use without the user.
- Rune Picker is Trade2-only: currency rewards price off the exchange; runes/essences/soul cores off the Trade2 exchange **book** (`refreshRuneBook`). Unbooked items show NOT FOUND (or "no price yet") and trigger a background book fill — next check / "Fetch fresh prices" shows the real price. Pending > fabricated (user's call). `fetchPrices` (blanked craft-pricer) and `fetchOptimizerMaterials` (experimental optimizer) were stubbed — they no longer fetch poe.ninja and return empty; rebuild on Trade2 if those tools come back.
- Illiquid high-value items (omens, Hinekora's): priced off the DIVINE side of the exchange via `divineMarketPrice` (cheapest *clustered* offer — robust to lowball bait + lone walls). Mirror isn't on the exchange (one price wall), so it's not sampled; would need Trade2 item-search to add.

## Gear Upgrade Finder — WORKING (PoB-driven, 2026-06-26)

## Conventions
- **ALWAYS push code changes immediately after making them — never leave them only local.** The VM deploys from `origin` (`git pull && docker compose up -d --build`), so an unpushed change is undeployable and the next session/the VM sees stale code. Edit → smoke-test → `git commit && git push` in the same session, every time. (This is the standing rule, not just a session-end nicety.)
- **Run `node smoke-test.js` after any UI/shell change.** 28 checks: pages/scripts parse, no @scope/iframe/token-cycle regressions, redirects, deep-link view init, 0 console errors, 0 overflow (desktop+mobile), a11y focus/announce, and the data-table styling (via mocked rows). Zero-dep static+HTTP checks always run; browser checks auto-find Playwright+Chromium (npx cache or global) and skip — never fail — if absent. Exit !=0 on failure. Proven to catch the deep-link regression (negative-control tested).
- **Browser verification must stay SILENT (no focus-stealing window) — but it's allowed.** The user wants full capability; they just don't want a window popping up and stealing focus from their game. Full `chrome.exe` — even headless — flashes a focus-stealing window on Windows; the **`chrome-headless-shell.exe`** binary is genuinely windowless. `smoke-test.js`'s `chromiumExe()` now PREFERS the headless-shell, so `node smoke-test.js` runs the browser checks silently — no window, no focus steal (verified 2026-06-24). So: prefer `curl`/API probes + reading code for most checks (faster), and use the browser when a visual/interaction check genuinely needs it — just make sure any Playwright launch uses the headless-shell `executablePath` (mirror `chromiumExe()`), never full `chrome.exe`. `SMOKE_NO_BROWSER=1 node smoke-test.js` still exists to skip browser checks entirely if ever needed, but it's no longer required.
- Keep every tool dependency-free and local-first.
- ALL trade2 API calls go through the shared persistent adaptive queue. Never call trade2 directly.
- **Trade2 testing discipline (don't over-limit yourself):** the user WANTS live verification, not skipped tests out of caution. Before any live Trade2 call, GET `/api/trade-status` and check `limited`/`secondsRemaining`/the `rateLimit` rule states FIRST; if clear, make the spaced live calls you need; if `limited`, wait the cooldown or use `previewOnly`. Check the limit, don't avoid the API. (See memory `dont-trade-limit-during-testing`.)
- **Record/replay = develop + test with ZERO live GGG calls (the legit answer to "testing trips the limit" — NOT IP rotation/evasion, which stays declined as ban-risky).** Implemented in `trade-queue.js` (the single chokepoint — ALL Trade2 incl. exchange/data/static goes through `tradeQueue.request`). `POE_RECORD=1 node server.js` → every real response is captured to `trade-fixtures.json` keyed by sha1(method+path+body). `POE_OFFLINE=1 node server.js` → serves those fixtures back, never touching the network or the rate-limit budget; `/api/trade-status` reports `replay:true, limited:false`; a fixture MISS throws (graceful `found:false` at the endpoints, never a silent live call). To capture fixtures: when the limit is quiet, `POE_RECORD=1 node server.js`, exercise the tool through the UI once (each unique request is recorded), stop, commit `trade-fixtures.json`. Then all later dev/smoke runs use `POE_OFFLINE=1` and never spend budget. The search `id` lives in the recorded search response, so a replayed search→fetch chain stays consistent. Self-check: `node trade-queue-test.js`.
- **NEVER raw-curl pathofexile.com — route EVERY call through `trade-queue.js`.** This includes the data endpoints `data/filters`, `data/items`, `data/stats` — they are rate-limited too and DON'T pre-warn. Raw `curl` bursts bypass the 3s spacing AND don't record the block in `.trade-status.json`, so they trip the shared IP and the queue can't even see it. ## Gotchas (landmines — read before touching pricing / market probes / UI)
- **The in-game price-check LIES for multi-mod items.** It prices off a handful of thin/aspirational *exact-match* listings, so it can read 10x+ the real liquid value. Verified live: juiced corrupted maps the overlay called "40–70ex" / "2 div" **sold for 5ex instantly**. Never quote the overlay as truth — trust the buy-side floor (cheapest real ask) or a confirmed sale.
- **Gate market probes on the ACTUAL item class.** A naive `sort:price asc` "cheapest at threshold" search returns junk (non-corrupted, low-roll, unjuiced) and badly under-prices. For waystones, gate on **corrupted (`misc_filters.corrupted.option:"true"`) + 0 revives (`map_filters.map_revives.max:0`)** to price the class the user actually has. This single mistake produced three wrong conclusions this session before it was caught.
- **Waystone value is COMBINATION-driven, not per-stat.** A single high stat is ~5ex bulk *even at max roll* (Monster Rarity ≥75 alone = 1–5ex, Drop ≥110 alone = 1ex). The chase is (a) Item Rarity ≥60% (~50–180ex, scarce; ≥70% doesn't exist), or (b) high-roll combos that all carry **Drop Chance ≥100%** (`monrar≥70+drop≥100+pack≥25` = 25/100/400ex). Per-stat curves can't express combos — `buildDump` keys on the combo signals, not a value sum. Drop ≥100% is also a real game breakpoint (guaranteed extra drops/pinnacle shards) and the user's "never trash >100%" keep-rule.
- **Rune book pricing — bid first, then a de-spammed ask floor (read this whole thing before touching `refreshRuneBook`/`cheapestAsk`).** The bulk exchange floods cheap items with PAR (1 ex : 1 item) asks and gives ZERO bids; it ALSO caps every response at ~100 listings TOTAL. Three real failures were fixed here (2026-06-24), each verified live:
  1. **Bid-only mispriced everything with sellers-but-no-bids as "no buyers."** The "of Aldur" runes have 14–35 sellers, zero bids; Cadigan's Epiphany (chase, ~500ex+) / Astrid's Creativity likewise. Fix: when no standing bid, price off the **cheapest de-spammed ask** (`side:"ask"`, source "trade2 exchange (ask)", "lowest ask — sell-side floor", confidence capped **low**).
  2. **`cheapestAsk` now drops par/sub-par floor spam (px ≤ 1 ex/item) then de-baits** (skip a lone ask under ½ the next — Cadigan's had a 1ex bait under a 4444ex wall). The book gate is then just **`ask.px ≥ 2`**. This is the spam guard: Greater Storm/Iron/Storm Rune are ALL 1ex par → filtered to nothing → stay "no buyers" (correct; sub-2ex items have no real exchange price). DO NOT reintroduce the old `stock≥2` skip (inflated thin chase items where stock-1 is the real floor) or the `aboveParCount≥3` gate (killed thin chase items with 1–2 sellers).
  3. **Page-cap starvation:** one spam whale (Greater Storm Rune = 99 of the 100-listing page, all 1ex) starves every other item in a shared batch to ZERO asks. Fix is **batched ask leg + starvation backfill** (`fetchExchangeChunked(…, RUNE_BOOK_BATCH, skipBackfill=false)`): one ~10-item page covers a whole reward paste, and any item the whale zeroed to no-offers gets re-fetched solo (`EXCHANGE_BACKFILL_CAP`). BOTH Rune Picker and Tab Tracker use this now. (2026-06-25: Rune Picker USED to fetch the ask leg PER ITEM via `perItemAsk → batchCap=1` — correct but it paid N calls × 3s EVERY check even when no whale was pasted, which was the "Rune Picker is slow" regression; reverted to batched+backfill. `perItemAsk` now only gates the item-search chase fallback, NOT ask batching.) Edge: partial starvation (whale leaves an item 1–2 offers, so it's "covered" but thin) won't trigger backfill and can under-price — rare for small reward pastes; re-check resolves it. The item-search fallback (Rune Picker only, budget 2/fill) still gives chase items their real price.
  - Book reads **`status:"any"`** (not online-only): chase items often have NO online seller, so online-only returned zero offers. Essences / genuinely sub-2ex runes still legitimately read "no buyers" — hand-curate if numbers are needed.
  4. **The bulk exchange CANNOT price chase items at all — use item-search.** Cadigan's Epiphany: in-game Market Ratio 576:1, but the exchange offer book is just a 1ex bait + a 4444ex wall (real trades clear instantly at the ratio and never sit as standing offers; GGG computes 576 from matched-trade history that the offer-search endpoint does NOT expose). So when the exchange ASK book is a thin wall (`cheapestAsk` now returns `depth` = # above-floor sellers; thin = `depth` 1–3), the Rune Picker fetches the price from the **item-search trade board** (`getTradePrice`, now de-baited + ex-converted) where chase items actually list and cluster. Gated: Rune Picker only (`perItemAsk`), `depth≥1` (something's really for sale), **budget 2 per fill** + stop-on-limited — because `/search`+`/fetch` is on a MUCH tighter rate policy than the exchange (a few in a row trip it and poison the whole fill). Booked `side:"market"`, source "trade2 search", confidence medium. Deep-book items (Astrid's `depth` 8 → 13ex) skip the search and use the cheap exchange ask. `RUNE_BOOK_VERSION`→10. (Cadigan's priced 310ex ≈ 1 divine — **confirmed correct by the user**; the "576" was a screenshot misread, so the divine→ex rate is fine, no separate bug.)
- **The Trade2 SEARCH endpoint (`/search` + `/fetch`) is on a MUCH tighter rate policy than the bulk exchange.** A few searches in a row trip it where the same number of exchange calls is fine.
